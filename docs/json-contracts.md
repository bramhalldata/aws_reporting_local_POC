# JSON Artifact Contracts

This document defines the stable JSON schemas for all artifacts produced by the publisher
and consumed by the portal. Schema changes must be reflected here before implementation.

---

## `run_history.json`

**Location:** `artifacts/current/run_history.json`
**Served at:** `/current/run_history.json`
**Envelope schema version:** `1.1.0`

The run history index is regenerated by the publisher on every successful run. It lists
all historical runs in descending order of `run_id` (most recent first).

### Envelope

```json
{
  "schema_version": "1.1.0",
  "generated_at": "2026-03-09T12:00:00+00:00",
  "runs": [ ... ]
}
```

| Field | Type | Description |
|-------|------|-------------|
| `schema_version` | string | Envelope format version (current: `"1.1.0"`) |
| `generated_at` | string | ISO 8601 UTC timestamp when the index was written |
| `runs` | array | List of run entries (see below) |

### Run Entry

```json
{
  "run_id": "20260309T120000Z",
  "dashboard_id": "dlq_operations",
  "report_ts": "2026-03-09T12:00:00Z",
  "generated_at": "2026-03-09T12:00:00+00:00",
  "status": "SUCCESS",
  "schema_version": "1.1.0",
  "artifacts": [
    { "name": "summary.json",    "type": "summary",    "path": "runs/20260309T120000Z/dlq_operations/summary.json" },
    { "name": "trend_30d.json",  "type": "trend_30d",  "path": "runs/20260309T120000Z/dlq_operations/trend_30d.json" },
    { "name": "top_sites.json",  "type": "top_sites",  "path": "runs/20260309T120000Z/dlq_operations/top_sites.json" },
    { "name": "exceptions.json", "type": "exceptions", "path": "runs/20260309T120000Z/dlq_operations/exceptions.json" }
  ]
}
```

| Field | Type | Description |
|-------|------|-------------|
| `run_id` | string | Compact ISO timestamp (e.g. `"20260309T120000Z"`) — uniquely identifies the run |
| `dashboard_id` | string | Dashboard identifier (e.g. `"dlq_operations"`, `"pipeline_health"`) |
| `report_ts` | string | ISO 8601 UTC timestamp anchoring all metric windows |
| `generated_at` | string | ISO 8601 UTC timestamp when artifacts were written |
| `status` | string | `"SUCCESS"` or `"FAILURE"` |
| `schema_version` | string | Dashboard artifact schema version (sourced from manifest) |
| `artifacts` | array | Structured artifact objects (see below) |

### Artifact Object (since schema v1.1.0)

```json
{ "name": "summary.json", "type": "summary", "path": "runs/20260309T120000Z/dlq_operations/summary.json" }
```

| Field | Type | Source | Description |
|-------|------|--------|-------------|
| `name` | string | manifest `artifacts[]` | Artifact filename (e.g. `"summary.json"`) |
| `type` | string | derived: `name.replace(".json", "")` | Stable role identifier for the artifact (e.g. `"summary"`, `"trend_30d"`) |
| `path` | string | computed: `runs/{run_id}/{dashboard_id}/{name}` | Relative path from `artifacts/` root. Prepend `/` for URL: `/<path>` |

**Notes:**
- `path` covers **historical run artifacts only** (under `runs/`). Current-run artifact
  paths are resolved by `useArtifactPath` in dashboard components (`/current/<dashboardId>/<filename>`).
- `type` is derived from the filename by stripping `.json`. A future phase may define it
  explicitly in `dashboard.json` if a richer taxonomy is needed.
- Optional future fields (`content_type`, `size_bytes`, `checksum`) may be added without
  breaking consumers of `name`, `type`, and `path`.

### Schema history

| Version | Change |
|---------|--------|
| `1.0.0` | Initial: `artifacts` was an array of bare filename strings |
| `1.1.0` | `artifacts` changed to structured objects with `name`, `type`, `path` |

## `manifest.json`

**Location:** `artifacts/current/<dashboardId>/manifest.json` and `artifacts/runs/<runId>/<dashboardId>/manifest.json`

The manifest is written by the publisher for each dashboard run. Its `artifacts` field
remains a **bare filename string list** — the enrichment to structured objects happens in
`_rebuild_run_history()` when assembling `run_history.json`.

```json
{
  "schema_version": "1.1.0",
  "run_id": "20260309T120000Z",
  "generated_at": "2026-03-09T12:00:00+00:00",
  "report_ts": "2026-03-09T12:00:00Z",
  "status": "SUCCESS",
  "artifacts": ["summary.json", "trend_30d.json", "top_sites.json", "exceptions.json"]
}
```

## Dashboard Artifact Files

Each dashboard produces a set of payload artifacts. Schemas are defined in
`src/publisher/validators/`. Current artifacts per dashboard:

### `dlq_operations`

| File | Schema | Description |
|------|--------|-------------|
| `summary.json` | `summary_schema.py` | KPI totals and top-sites rollup |
| `trend_30d.json` | `trend_30d_schema.py` | 30-day daily failure counts |
| `top_sites.json` | `top_sites_schema.py` | Top sites by failures over 30 days |
| `exceptions.json` | `exceptions_schema.py` | Failure types over last 7 days |

### `pipeline_health`

| File | Schema | Description |
|------|--------|-------------|
| `summary.json` | `pipeline_health_summary_schema.py` | Document and site KPIs |
| `failure_types.json` | `pipeline_health_failure_types_schema.py` | Failure type breakdown |
