# JSON Artifact Contracts

This document defines the stable JSON schemas for all artifacts produced by the publisher
and consumed by the portal. Schema changes must be reflected here before implementation.

---

## `run_history.json`

**Location:** `artifacts/{client}/{env}/current/run_history.json`
**Served at:** `/{client}/{env}/current/run_history.json`
**Envelope schema version:** `1.2.0`

The run history index is regenerated by the publisher on every successful run. It lists
all historical runs in descending order of `run_id` (most recent first).

### Envelope

```json
{
  "schema_version": "1.2.0",
  "client_id": "default",
  "env_id": "local",
  "generated_at": "2026-03-09T12:00:00+00:00",
  "runs": [ ... ]
}
```

| Field | Type | Description |
|-------|------|-------------|
| `schema_version` | string | Envelope format version (current: `"1.2.0"`) |
| `client_id` | string | Client identifier (e.g. `"default"`, `"contexture"`) |
| `env_id` | string | Environment identifier (e.g. `"local"`, `"prod"`) |
| `generated_at` | string | ISO 8601 UTC timestamp when the index was written |
| `runs` | array | List of run entries (see below) |

### Run Entry

```json
{
  "run_id": "20260309T120000Z",
  "dashboard_id": "dlq_operations",
  "report_ts": "2026-03-09T12:00:00Z",
  "generated_at": "2026-03-09T12:00:00+00:00",
  "status": "SUCCESS",
  "schema_version": "1.2.0",
  "artifacts": [
    { "name": "summary.json",    "type": "summary",    "path": "default/local/runs/20260309T120000Z/dlq_operations/summary.json" },
    { "name": "trend_30d.json",  "type": "trend_30d",  "path": "default/local/runs/20260309T120000Z/dlq_operations/trend_30d.json" },
    { "name": "top_sites.json",  "type": "top_sites",  "path": "default/local/runs/20260309T120000Z/dlq_operations/top_sites.json" },
    { "name": "exceptions.json", "type": "exceptions", "path": "default/local/runs/20260309T120000Z/dlq_operations/exceptions.json" }
  ]
}
```

| Field | Type | Description |
|-------|------|-------------|
| `run_id` | string | Compact ISO timestamp (e.g. `"20260309T120000Z"`) — uniquely identifies the run |
| `dashboard_id` | string | Dashboard identifier (e.g. `"dlq_operations"`, `"pipeline_health"`) |
| `report_ts` | string | ISO 8601 UTC timestamp anchoring all metric windows |
| `generated_at` | string | ISO 8601 UTC timestamp when artifacts were written |
| `status` | string | `"SUCCESS"` or `"FAILURE"` |
| `schema_version` | string | Dashboard artifact schema version (sourced from manifest) |
| `artifacts` | array | Structured artifact objects (see below) |

### Artifact Object (since schema v1.1.0)

```json
{ "name": "summary.json", "type": "summary", "path": "default/local/runs/20260309T120000Z/dlq_operations/summary.json" }
```

| Field | Type | Source | Description |
|-------|------|--------|-------------|
| `name` | string | manifest `artifacts[]` | Artifact filename (e.g. `"summary.json"`) |
| `type` | string | derived: `name.replace(".json", "")` | Stable role identifier for the artifact (e.g. `"summary"`, `"trend_30d"`) |
| `path` | string | computed by publisher: `{client_id}/{env_id}/runs/{run_id}/{dashboard_id}/{name}` | Relative path from `artifacts/` root. Prepend `/` for URL: `/<path>` |

**Notes:**
- `path` covers **historical run artifacts only** (under `runs/`). Current-run artifact
  paths are resolved by `useArtifactPath` in dashboard components (`/{client}/{env}/current/{dashboardId}/{filename}`).
- **Portal never constructs artifact paths** — `path` is publisher-owned. `RunDetail` renders
  `href={/${artifact.path}}` directly without modification.
- `type` is derived from the filename by stripping `.json`. A future phase may define it
  explicitly in `dashboard.json` if a richer taxonomy is needed.
- Optional future fields (`content_type`, `size_bytes`, `checksum`) may be added without
  breaking consumers of `name`, `type`, and `path`.

### Schema history

| Version | Change |
|---------|--------|
| `1.0.0` | Initial: `artifacts` was an array of bare filename strings |
| `1.1.0` | `artifacts` changed to structured objects with `name`, `type`, `path` |
| `1.2.0` | Envelope gains required `client_id` and `env_id`; `artifact.path` gains `{client_id}/{env_id}/` prefix |

## `manifest.json`

**Location:** `artifacts/{client}/{env}/current/<dashboardId>/manifest.json` and `artifacts/{client}/{env}/runs/<runId>/<dashboardId>/manifest.json`

The manifest is written by the publisher for each dashboard run. Its `artifacts` field
remains a **bare filename string list** — the enrichment to structured objects happens in
`_rebuild_run_history()` when assembling `run_history.json`.

```json
{
  "schema_version": "1.2.0",
  "run_id": "20260309T120000Z",
  "generated_at": "2026-03-09T12:00:00+00:00",
  "report_ts": "2026-03-09T12:00:00Z",
  "status": "SUCCESS",
  "artifacts": ["summary.json", "trend_30d.json", "top_sites.json", "exceptions.json"]
}
```

## Dashboard Artifact Files

Each dashboard produces a set of payload artifacts. Schemas are defined in
`src/publisher/validators/`. Current artifacts per dashboard:

### `dlq_operations`

| File | Schema | Description |
|------|--------|-------------|
| `summary.json` | `summary_schema.py` | KPI totals and top-sites rollup |
| `trend_30d.json` | `trend_30d_schema.py` | 30-day daily failure counts |
| `top_sites.json` | `top_sites_schema.py` | Top sites by failures over 30 days |
| `exceptions.json` | `exceptions_schema.py` | Failure types over last 7 days |

### `pipeline_health`

| File | Schema | Description |
|------|--------|-------------|
| `summary.json` | `pipeline_health_summary_schema.py` | Document and site KPIs |
| `failure_types.json` | `pipeline_health_failure_types_schema.py` | Failure type breakdown |

### `sent_to_udm`

| File | Schema | Description |
|------|--------|-------------|
| `summary.json` | `sent_to_udm_summary_schema.py` | Lifetime and 30-day KPI totals |
| `region_summary.json` | `sent_to_udm_region_summary_schema.py` | Per-region CCD counts, site counts, and date range |
| `trend_30d.json` | `sent_to_udm_trend_30d_schema.py` | 30-day daily CCD sent counts (aggregate) |
| `lifetime_detail.json` | `sent_to_udm_lifetime_detail_schema.py` | Per-site lifetime CCD counts and date range |
| `recent_detail_30d.json` | `sent_to_udm_recent_detail_30d_schema.py` | Per-site CCD counts within the last 30-day window |

#### `summary.json`

```json
{
  "schema_version": "1.2.0",
  "generated_at": "2026-03-17T18:51:42+00:00",
  "report_ts": "2026-03-17T18:51:42Z",
  "total_regions_active": 4,
  "total_sites_active": 6,
  "total_ccds_sent": 300,
  "earliest_event_ts": "2026-01-06 23:59:14-05:00",
  "latest_event_ts": "2026-03-07 01:07:46-05:00",
  "regions_active_30d": 4,
  "sites_active_30d": 6
}
```

| Field | Type | Description |
|-------|------|-------------|
| `total_regions_active` | integer | Distinct regions with at least one CCD sent (all time) |
| `total_sites_active` | integer | Distinct sites with at least one CCD sent (all time) |
| `total_ccds_sent` | integer | Total CCD count across all regions and sites (all time) |
| `earliest_event_ts` | string | `MIN(timestamp)` across all CCD events — data-derived, not `report_ts`-relative |
| `latest_event_ts` | string | `MAX(timestamp)` across all CCD events — data-derived, not `report_ts`-relative |
| `regions_active_30d` | integer | Distinct regions active within the last 30 days of `report_ts` |
| `sites_active_30d` | integer | Distinct sites active within the last 30 days of `report_ts` |

#### `region_summary.json`

```json
{
  "schema_version": "1.2.0",
  "generated_at": "2026-03-17T18:51:42+00:00",
  "report_ts": "2026-03-17T18:51:42Z",
  "regions": [
    { "region": "AZ", "site_count": 2, "ccd_count": 87,
      "first_seen": "2026-01-06 23:59:14-05:00",
      "last_seen": "2026-03-07 01:07:46-05:00" }
  ]
}
```

| Field | Type | Description |
|-------|------|-------------|
| `regions[].region` | string | Region identifier |
| `regions[].site_count` | integer | Number of distinct sites in this region (all time) |
| `regions[].ccd_count` | integer | Total CCDs sent from this region (all time) |
| `regions[].first_seen` | string | Earliest CCD timestamp for this region (all time) |
| `regions[].last_seen` | string | Most recent CCD timestamp for this region (all time) |

#### `trend_30d.json`

```json
{
  "schema_version": "1.2.0",
  "generated_at": "2026-03-17T18:51:42+00:00",
  "report_ts": "2026-03-17T18:51:42Z",
  "days": [
    { "date": "2026-02-16", "ccd_count": 4 }
  ]
}
```

**Note:** This artifact uses `ccd_count` — not `failures`. The filename `trend_30d.json` is shared with
`dlq_operations`, but the schemas are not interchangeable. Each artifact is scoped to its dashboard
directory. Days with no CCD activity are included with `ccd_count: 0` (zero-filled via date spine).

#### `lifetime_detail.json`

```json
{
  "schema_version": "1.2.0",
  "generated_at": "2026-03-17T18:51:42+00:00",
  "report_ts": "2026-03-17T18:51:42Z",
  "rows": [
    { "region": "AZ", "site": "az_site_1", "ccd_count": 87,
      "first_seen": "2026-01-06 23:59:14-05:00",
      "last_seen": "2026-03-07 01:07:46-05:00" }
  ]
}
```

| Field | Type | Description |
|-------|------|-------------|
| `rows[].region` | string | Region identifier |
| `rows[].site` | string | Site identifier (maps to "Facility" in portal display) |
| `rows[].ccd_count` | integer | Total CCDs sent from this site (all time) |
| `rows[].first_seen` | string | Earliest CCD timestamp for this site (all time) |
| `rows[].last_seen` | string | Most recent CCD timestamp for this site (all time) |

#### `recent_detail_30d.json`

```json
{
  "schema_version": "1.2.0",
  "generated_at": "2026-03-17T18:51:42+00:00",
  "report_ts": "2026-03-17T18:51:42Z",
  "window_days": 30,
  "rows": [
    { "region": "AZ", "site": "az_site_1", "ccd_count": 21,
      "first_seen_30d": "2026-02-15 18:20:23-05:00",
      "last_seen_30d": "2026-03-05 08:51:02-05:00" }
  ]
}
```

| Field | Type | Description |
|-------|------|-------------|
| `window_days` | integer | Size of the rolling window (always `30`) |
| `rows[].region` | string | Region identifier |
| `rows[].site` | string | Site identifier |
| `rows[].ccd_count` | integer | CCDs sent from this site within the window |
| `rows[].first_seen_30d` | string | Earliest CCD timestamp within the window. Note the `_30d` suffix — distinguishes from `lifetime_detail.json`'s `first_seen` |
| `rows[].last_seen_30d` | string | Most recent CCD timestamp within the window. Note the `_30d` suffix — distinguishes from `lifetime_detail.json`'s `last_seen` |
