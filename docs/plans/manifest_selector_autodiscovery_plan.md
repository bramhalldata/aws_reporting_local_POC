# Manifest-Driven Selector Auto-Discovery — Implementation Plan

**Feature:** Manifest-Driven Selector Auto-Discovery
**Output artifact:** `docs/plans/manifest_selector_autodiscovery_plan.md`
**Date:** 2026-03-10
**Status:** Draft — pending review

---

## Context

`portal/src/config/scopes.js` is a static list that must be manually maintained.
When a new scope is bootstrapped via `publisher bootstrap --client X --env Y`,
the selector dropdown does not show it until a developer edits `scopes.js`.

The platform manifest (`artifacts/platform-manifest.json`) is now generated
automatically after every `publisher run()`. It is served at `/platform-manifest.json`
via Vite `publicDir: ../artifacts`. It contains the complete, live list of bootstrapped
client/env scopes with their dashboards.

This feature replaces `scopes.js` with a fetch of `/platform-manifest.json` so the
selector auto-discovers all bootstrapped scopes without manual configuration.

---

## 1. Feature Summary

| Benefit | Description |
|---------|-------------|
| Zero manual config | Bootstrapping a new scope automatically makes it appear in the selector |
| Single source of truth | Manifest is already generated by the publisher — no duplicate maintenance |
| Fallback safety | While loading or on error, selector falls back to the current URL scope |
| No new infrastructure | Manifest is already served at `/platform-manifest.json` via Vite publicDir |

## 2. Manifest Schema Reference

```json
{
  "schema_version": "1.0.0",
  "generated_at": "2026-03-10T18:11:11+00:00",
  "clients": [
    {
      "client_id": "default",
      "envs": [
        { "env_id": "local", "dashboards": [...] }
      ]
    },
    {
      "client_id": "contexture",
      "envs": [
        { "env_id": "local", "dashboards": [...] }
      ]
    }
  ]
}
```

The relevant transform for the selector is:

```js
manifest.clients.map(c => ({
  client: c.client_id,
  envs:   c.envs.map(e => e.env_id),
}))
```

This produces the same shape as `SCOPES` in `scopes.js` — `resolveEnv()` and
`targetUrl()` in `selectorNav.js` work without modification.

## 3. New Hook: `usePlatformManifest.js`

**Location:** `portal/src/hooks/usePlatformManifest.js`

Following the established hook pattern (`useArtifactPath.js`): single-purpose,
no side effects beyond the fetch.

```js
import { useState, useEffect } from "react";

export function usePlatformManifest() {
  const [scopes, setScopes] = useState(null); // null = still loading
  const [error,  setError]  = useState(null);

  useEffect(() => {
    fetch("/platform-manifest.json")
      .then(res => {
        const ct = res.headers.get("content-type") ?? "";
        if (!res.ok || !ct.includes("application/json"))
          throw new Error("platform-manifest.json not available");
        return res.json();
      })
      .then(manifest => {
        const derived = (manifest.clients ?? []).map(c => ({
          client: c.client_id,
          envs:   c.envs.map(e => e.env_id),
        }));
        setScopes(derived);
      })
      .catch(err => {
        console.warn("[usePlatformManifest] fetch failed:", err.message);
        setError(err.message);
      });
  }, []);

  return { scopes, loading: scopes === null && error === null, error };
}
```

**Design notes:**

- `scopes === null` with `error === null` means loading is in progress
- `scopes = []` means manifest was fetched but has no clients (valid — empty platform)
- Content-type guard: Vite SPA fallback returns `text/html` with HTTP 200 for missing files; same pattern used in `RunHistory.jsx`, `RunDetail.jsx`, and dashboard components
- Fetches once on mount; no polling — page refresh is sufficient to pick up new scopes
- `clients ?? []` guard: safe against malformed manifest (missing clients key)

## 4. Modified: `ClientEnvSelector.jsx`

**Current imports:**

```js
import { SCOPES } from "../config/scopes.js";
```

**New imports (replace the above):**

```js
import { usePlatformManifest } from "../hooks/usePlatformManifest.js";
```

**Inside the component — replace SCOPES usage:**

```js
export default function ClientEnvSelector() {
  const { client, env } = useParams();
  const { scopes }      = usePlatformManifest();
  const navigate        = useNavigate();
  const location        = useLocation();

  // While loading or on error: fall back to the current URL scope only.
  // Selector remains functional (current scope is always shown).
  const effectiveScopes = scopes ?? [{ client, envs: [env] }];

  const clients      = effectiveScopes.map(s => s.client);
  const currentEntry = effectiveScopes.find(s => s.client === client)
    ?? { client, envs: [env] };  // guard: current client not yet in manifest
  const envs = currentEntry.envs;

  // handleClientChange and handleEnvChange: unchanged
  // — they use resolveEnv() and targetUrl() which accept the same scopes shape
  ...
}
```

**What does NOT change:**

- `handleClientChange` — calls `resolveEnv(newClientEntry, env)` → unchanged
- `handleEnvChange` — calls `targetUrl()` → unchanged
- All styles — unchanged
- JSX structure — unchanged
- `dashboardIds` derivation from `dashboardMeta` — unchanged

## 6. Loading and Error Behavior

| State | Selector behavior |
|-------|------------------|
| Loading (manifest fetch in flight) | Shows only current scope (from URL params); functional |
| Fetch error (network/missing file) | Shows only current scope; console.warn logged |
| Fetch success | Shows all scopes from manifest |
| Current client not in manifest | Falls back to `{ client, envs: [env] }` guard |

No error UI is shown in the selector. The fallback is silent and functional.
This is appropriate for a developer tool where the console is monitored.

## 9. Performance

The manifest fetch is a single HTTP request on selector mount. File size is O(clients × envs):
currently ~800 bytes. Vite dev server serves it from the filesystem; CloudFront would cache it.
No concern at any foreseeable scale.

## 11. Non-Goals

| Excluded | Reason |
|----------|--------|
| Polling / live refresh during session | Page refresh is sufficient |
| Global React Context for manifest | Only needed by `ClientEnvSelector` |
| Tests for `usePlatformManifest` | Thin fetch wrapper; covered by E2E verification |
| Error UI in the selector | Silent fallback is appropriate |
| Scoped manifests | Global manifest is sufficient |
